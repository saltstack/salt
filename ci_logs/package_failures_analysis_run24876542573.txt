2) Root cause (Linux rows sampled)
   Amazon Linux 2023 ``install``, ``upgrade``, ``downgrade`` jobs and Debian 12
   ``install`` all reported a single failing testcase:

     tests.pytests.pkg.integration.test_salt_user::test_pkg_paths
     AssertionError: assert 'root' == 'salt'

   The old implementation used ``os.walk`` over ``/etc/salt``, ``/var/cache/salt``,
   etc., and assumed any directory not on the explicit salt-user list was
   ``root:root``. Current packages add legitimate ``root`` subtrees (cache/proc
   style paths) while parent paths still participate in salt-owned checks via
   ``salt_user_subdirs``, which made the walk inconsistent.

3) Code changes (not committed / not pushed)
   a) tests/pytests/pkg/integration/test_salt_user.py
      - ``test_pkg_paths`` now:
        * Asserts each top-level ``pkg_paths`` directory exists and is ``root:root``.
        * Asserts each path in ``pkg_paths_salt_user`` (minus exclusions) that
          exists on disk is owned by ``salt:salt``.
      - Removed unused ``os`` import and the unused ``salt_call_cli`` fixture arg.

   b) tools/testsuite/download.py
      - Broadened the package artifact glob to
        ``salt-*-{arch}-{pkg_type}*`` so CI names like
        ``salt-3007.13+97.g06163540c8-x86_64-rpm-from-src`` match (required for
        ``tools ts download test-artifacts`` against this run).

5) macOS package matrix (Test Package / macOS 15 … upgrade|downgrade …)
   JUnit + logs from run 24876542573 showed ``FactoryNotStarted`` on the packaged
   ``SaltMaster`` while the master had already logged and forwarded the
   ``salt/master/.../start`` event — salt-factories treated naive ``_stamp`` wall
   times as UTC. Fix: ``tests/pytests/pkg/conftest.py`` replaces
   ``_convert_stamp`` on Darwin and bumps ``start_timeout`` for Darwin masters.
   Reproduced green on a macOS 14 x86_64 host (SSH ``localhost -p 2222``) with
   ``ci-test-onedir-pkgs`` ``upgrade`` / ``downgrade`` slices and
   ``--prev-version=3007.13``.
